# 差距分析

## 1. 分析范围

本分析基于以下输入：

- 参考仓库 `Johnixr/agent-cli-guide` 的 10 条 CLI 设计原则
- 本仓库当前 `spec/`、`docs/`、`issues-plan/` 文档体系
- 当前代码结构、命令帮助、I/O 和交互实现抽样

本分析只用于制定改进方案，不等同于立即承诺全部进入下一版本。

## 2. 当前项目的优势

### 2.1 命令树结构已经具备 agent 友好基础

当前 CLI 主体采用资源分组方式：

- `auth`
- `repo`
- `issue`
- `pr`
- `commit`
- `label`
- `milestone`
- `release`

这与参考方案强调的 `noun -> verb` 结构基本一致，是后续继续增强 agent 友好性的最大基础。

### 2.2 已经存在一些关键能力雏形

当前仓库并非从零开始，已经具备一些可复用基础：

- 有 `IOStreams` 抽象，并具备 TTY / 颜色检测
- 已支持 `NO_COLOR` / `TERM=dumb`
- 多个危险命令已经支持 `--yes`
- 部分命令已经支持 `--web`
- 少量命令已有 `--json`
- `docs/COMMANDS.md` 已作为命令行为真相源

问题不在于完全没有基础，而在于这些能力还没有统一成框架和契约。

## 3. 主要差距

## 3.1 输出契约差距最大

参考方案把“结构化输出”视为 CLI 的稳定 API 契约；当前项目在这一点上明显不足。

现状判断：

- 当前 `pkg/cmd` 中约有 68 个 `NewCmd*` 命令构造函数
- 显式定义 `--json` 的命令只有 2 个：`issue comments`、`commit view`
- 多数命令仍直接输出面向人类阅读的文本
- `stdout` / `stderr` 的职责没有形成统一规则

影响：

- 代理难以稳定解析数据
- 同一资源在不同命令中的输出风格不一致
- 后续引入脚本、MCP、外部自动化时，复用成本高

这意味着当前 CLI 更像“人用命令集合”，还不是“可被代理稳定编排的命令接口”。

## 3.2 TTY / 非 TTY 行为只做了检测，没有形成统一政策

当前已有 TTY 检测能力，但尚未统一贯彻到命令行为。

现状判断：

- `pkg/iostreams/iostreams.go` 已提供 `IsStdinTTY`、`IsStdoutTTY`、`IsStderrTTY`
- 颜色控制已有基础能力
- 但绝大多数命令不会因为非 TTY 自动切到 JSON 或关闭交互
- 若命令需要输入，当前更多依赖命令内部自行读取 stdin

影响：

- 代理在 pipe / 非交互环境下的行为不可预测
- 未来新增交互命令时，容易出现“非 TTY 卡住等待输入”
- CLI 使用者和自动化调用者看到的是两套未定义的隐式行为

## 3.3 交互确认机制不统一，存在阻塞风险

破坏性操作虽然已有确认保护，但实现方式仍较原始。

现状判断：

- `repo delete`、`label delete`、`milestone delete`、`release delete` 支持 `--yes`
- 这些命令内部仍有 `fmt.Scanln(...)` 一类直接读取输入的实现
- 部分输入路径直接通过 `ReadString('\n')` 读取 stdin
- 尚未建立统一的“无 TTY 时禁止交互、缺输入时立即失败”规则

影响：

- 行为难以统一测试
- 不利于未来把确认机制扩展为统一组件
- 代理调用破坏性命令时，缺少稳定的非交互保障

## 3.4 错误模型与退出码缺少稳定语义

参考方案非常强调“机器可消费的错误”和“语义化退出码”，当前项目在这方面仍是初级状态。

现状判断：

- 根命令在入口仍统一 `os.Exit(1)`
- 业务命令大多返回字符串错误
- API 层虽有 `APIError` 结构，但没有上升为 CLI 统一错误模型
- 文档中没有稳定的退出码定义

影响：

- 代理无法通过退出码快速判断“参数错 / 权限错 / 不存在 / 冲突 / 可重试”
- 调用方只能依赖字符串匹配做恢复逻辑
- 后续要做更高阶自动化时，稳定性不够

## 3.5 破坏性与写操作缺少统一 dry-run / 幂等策略

这是当前与参考方案差距第二大的部分。

现状判断：

- 仓库中没有 `--dry-run`
- 没有 schema 化的“预执行结果”
- 没有统一的冲突退出码和 `--if-not-exists` / `--if-exists`
- 创建类命令多为一次性写操作语义

影响：

- 代理做写操作前无法低成本确认后果
- 重试风险高，尤其对 create / comment / review 一类命令
- 一旦未来接入更高自动化层，会限制可恢复性和可编排性

## 3.6 Help 文本可读，但还不是高质量“发现接口”

当前大部分命令有 `Short` / `Long` / `Example`，这是优点；但离“高质量 agent help”还有明显差距。

现状判断：

- 帮助文本通常有示例
- 但没有统一强调必填 / 选填 / 值域 / JSON 输出 / 非交互语义
- 同类命令在 flag 命名和例子密度上并不完全一致
- 帮助文本没有作为产品契约被治理

影响：

- 代理虽然能探索命令树，但仍需要靠猜测补齐参数语义
- 文档真相源在 `docs/COMMANDS.md`，但 `--help` 还没有达到同等级别的可靠度

## 3.7 缺少 schema introspection 能力

参考方案建议提供类似 `mytool schema` 的命令，让代理按需查询参数结构；当前项目完全没有这类能力。

影响：

- 代理需要在大块帮助文本和文档之间手工提取规则
- 一旦命令数量增长，探索成本会变高
- 后续若要和 MCP、编辑器插件、自动化平台联动，将缺少统一的命令元数据出口

## 3.8 文档治理已成熟，但“agent-friendly CLI”尚未入规

当前项目在治理层其实比很多仓库成熟：

- `spec/`、`docs/`、`issues-plan/` 分层清晰
- AI 入口和 skill 适配层已经分离
- 当前也在做真实行为校准

但问题在于：

- 这些治理文件还没有单独定义 agent-friendly CLI 的能力边界
- 命令输出契约、错误契约、help 质量、schema 能力仍未进入正式规范

结论：

项目的治理框架已经准备好了，真正缺的是“把代理友好性纳入正式规范和交付门禁”。

## 4. 差距优先级判断

建议按以下顺序收口：

### P0：统一契约层

- 统一输出格式与 `--json`
- 统一非 TTY 行为
- 统一错误模型和退出码

这是 agent 可稳定调用 CLI 的前提。

### P1：统一安全写操作语义

- `--dry-run`
- `--yes` / `--no-interactive`
- 幂等参数和冲突语义

这是从“可读”走向“可执行”的关键步骤。

### P2：统一发现层

- `schema` 命令
- `--help` 模板化
- 文档与 skill 同步增强

这是从“能用”走向“规模化被代理消费”的关键。

## 5. 总判断

`gitcode-cli` 不需要推倒重来。

更合理的路线是：

1. 保留现有资源树结构和模块划分
2. 在 `cmdutil` / `iostreams` / 错误处理层补一层统一契约
3. 优先改高频读命令和高风险写命令
4. 再把帮助、schema、skills 和文档一起补齐

因此，本项目最适合的不是“大重构”，而是一次“面向 agent 消费的契约化改造”。
