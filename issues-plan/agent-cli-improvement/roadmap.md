# 分阶段路线图

## 1. 路线原则

本路线图遵循四个原则：

- 先统一基座，再批量迁移命令
- 先覆盖高频读命令，再覆盖高风险写命令
- 先做可验证契约，再做体验增强
- 每一阶段都必须同步 `spec/`、`docs/`、AI 入口和 skills

## 2. 阶段划分

## 阶段 A：专项立项与规范入轨

目标：

- 把“agent-friendly CLI”从参考建议转成项目正式改进主题
- 建立 roadmap、里程碑和 issue 拆分
- 定义正式规范落点

交付：

- 本目录下的方案文档
- 新 milestone
- 一组分解 issue
- 后续要新增的规范文档清单

验收：

- 方案边界清晰
- issue 能直接进入执行
- 与现有 `spec/` / `docs/` / `issues-plan/` 不冲突

## 阶段 B：统一基础契约

目标：

- 建立统一输出、交互、错误和退出码基座

建议交付：

- 新增 agent-friendly CLI 规范文档
- 统一输出模式和错误模型
- 增加退出码文档
- 抽象统一 confirmation / non-interactive 组件

验收：

- 新增共享运行时能力
- 至少有 1 组核心命令接入新能力
- 非 TTY 行为和错误模型有正式文档

## 阶段 C：高频读命令迁移

目标：

- 让最常被查询的命令具备稳定结构化输出

建议优先命令：

- `repo view`
- `repo list`
- `issue list`
- `issue view`
- `pr list`
- `pr view`
- `release list`
- `release view`

验收：

- 这些命令支持统一 `--json`
- 文本和 JSON 输出职责清晰
- 帮助文本明确标注输出能力

## 阶段 D：高风险写命令迁移

目标：

- 解决交互阻塞、安全确认和写操作可恢复性问题

建议优先命令：

- `repo delete`
- `release delete`
- `label delete`
- `milestone delete`
- `issue create`
- `pr create`
- `pr review`

验收：

- 删除类命令统一确认组件
- 非 TTY 缺输入时明确失败
- 第一批写命令支持 `--dry-run` 或明确预检查模式

## 阶段 E：发现层与生态层补齐

目标：

- 让代理能够更低成本探索命令能力

建议交付：

- `gc schema`
- 帮助模板统一
- `docs/COMMANDS.md` 结构优化
- AI skill 更新
- 回归矩阵扩展到契约层测试

验收：

- 代理无需阅读整本文档也能查询命令模式
- 新旧命令帮助一致性明显提升
- 契约层行为可被回归验证

## 3. 优先级建议

## 必做

- 规范入轨
- 统一输出契约
- 统一非交互行为
- 统一错误与退出码
- 删除类命令统一确认机制

## 应做

- 高频读命令 JSON 化
- 第一批写命令 dry-run / 预检查
- help 模板统一

## 可后置

- 全量命令 JSON 覆盖
- 全量幂等参数
- 更复杂 schema / 生态集成

## 4. 风险与对策

### 风险 1：一次性改动面过大

对策：

- 共享基座先行
- 命令分批迁移
- 以命令组为单位分 issue

### 风险 2：文本输出兼容性被打破

对策：

- TTY 默认维持文本
- JSON 能力先以新增 flag 方式进入
- 非 TTY 默认 JSON 作为后续显式升级项

### 风险 3：文档与代码再次漂移

对策：

- 把 agent-friendly CLI 写进正式规范和质量门禁
- 所有 issue 都带文档同步验收项

### 风险 4：dry-run 难以一次做完整

对策：

- 允许“真实 dry-run”和“预检查模式”分层推进
- 优先在 delete / create 等高风险命令试点

## 5. 里程碑建议

建议建立新 milestone：

- 名称：`v0.6.0 Agent-Friendly CLI`

建议描述：

- 以 `agent-cli-guide` 为参考，对 `gitcode-cli` 进行一轮面向 AI 代理消费的契约化改造，重点覆盖结构化输出、非 TTY 行为、错误与退出码、写操作安全语义、帮助文本和 schema 发现能力。

建议计划窗口：

- 起始：2026-03-30
- 建议完成目标：2026-05-15

这个 milestone 不意味着必须在一个版本里做完整闭环，但足以作为下一轮工程治理主题。
