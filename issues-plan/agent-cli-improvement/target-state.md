# 目标状态

## 1. 总体目标

将 `gitcode-cli` 从“功能可用的人工 CLI”升级为“同时面向人类与 AI 代理稳定可消费的 CLI”。

这个目标不要求所有命令一次性完全重做，而是要求建立一套统一能力底座，使新老命令可以逐步迁移。

## 2. 目标能力模型

建议最终目标包含 6 层。

### 2.1 命令结构层

保持当前资源树结构，不做破坏性改名：

- `gc repo <verb>`
- `gc issue <verb>`
- `gc pr <verb>`
- `gc release <verb>`
- `gc milestone <verb>`
- `gc commit <verb>`

补充要求：

- 同类动作统一 verb 命名
- 不引入同义动作漂移
- 新增命令优先复用既有 verb

### 2.2 输出契约层

建立统一输出模型：

- 所有有返回数据的命令最终都支持 `--json`
- 非 TTY 默认优先结构化输出
- `stdout` 只输出数据
- `stderr` 输出提示、警告、进度、错误

建议能力：

- 输出模式统一为 `text` / `json`
- 列表型命令后续可扩展 `ndjson`
- JSON 字段变更纳入兼容性规则

### 2.3 交互与执行环境层

建立统一的执行行为基线：

- 非 TTY 默认禁用交互确认
- 缺少必填输入时明确报错，不允许挂起等待
- `--yes` 跳过全部确认
- 引入 `--no-interactive` 作为全局或统一 flag
- 浏览器打开、分页器、彩色输出都受 TTY 和显式 flag 控制

### 2.4 错误与退出码层

建立统一错误模型：

- CLI 错误输出支持结构化格式
- 错误包含类型、说明、建议、是否可重试
- 退出码定义正式文档

建议最小退出码集合：

- `0` 成功
- `1` 通用错误
- `2` 参数或用法错误
- `3` 资源不存在
- `4` 权限或认证错误
- `5` 资源冲突
- `10` dry-run 成功

### 2.5 安全写操作层

所有写路径逐步具备统一保护：

- `create` / `edit` / `delete` / `merge` / `review` 等命令可判定是否支持 `--dry-run`
- 不适合真实 dry-run 的命令至少给出预检查或 explain 模式
- 破坏性命令统一确认组件
- Create / delete 场景逐步引入幂等参数

### 2.6 发现与文档层

把“帮助文本 + schema + docs + skills”作为同一产品面治理：

- `--help` 模板统一
- 增加 `gc schema` 或 `gc meta schema`
- `docs/COMMANDS.md` 与 `--help` 保持同源策略
- `.ai/skills/`、`.codex/skills/`、`.claude/skills/` 同步 agent-friendly 使用建议

## 3. 推荐架构落点

## 3.1 在 `cmdutil` 增加统一运行时选项

建议引入统一运行时能力，而不是每个命令自行实现：

- 输出模式解析
- 非 TTY 默认策略
- 错误包装与退出码映射
- 写操作 dry-run / confirmation 策略
- 通用帮助模板

这部分应成为后续所有命令的共享基础设施。

## 3.2 在 `iostreams` 扩展环境感知能力

当前 `iostreams` 已有基础能力，建议继续扩展：

- 非 TTY 默认行为判定
- 输出目标能力标记
- 统一颜色 / 分页 / 交互开关
- 提供“当前命令是否允许 prompt”的单一判断入口

## 3.3 在 `api` 与 CLI 之间增加错误归一层

API 已能解析部分错误，但还缺少 CLI 级归一：

- HTTP / API 错误映射为统一错误类型
- 统一 recovery suggestion
- 统一 retryable 语义
- 统一退出码映射

## 3.4 对命令迁移采用“基座先行、命令分批接入”

不建议一次性改 68 个命令。

建议迁移顺序：

1. 高频只读命令
2. 高风险删除类命令
3. 高频创建 / 编辑命令
4. 低频辅助命令

## 4. 兼容性策略

这是本专项能否落地的关键。

### 4.1 文本输出默认尽量兼容

在 TTY 场景下，优先保留用户熟悉的文本输出，避免一次性打断人工使用习惯。

### 4.2 JSON 输出作为新增契约

新增 `--json`、`schema`、结构化错误属于增强，不应破坏既有文本使用路径。

### 4.3 非 TTY 行为变化需要显式公告

如果未来决定“非 TTY 默认 JSON”，必须：

- 先形成正式规范
- 更新 `docs/COMMANDS.md`
- 更新 README 和 AI 入口文档
- 在 release notes 中明确说明

### 4.4 退出码变更视为兼容性事件

退出码一旦文档化，就要按兼容性规则治理，不应随意漂移。

## 5. 验收标准

当本专项达到首轮收口时，至少应满足：

- 一组核心高频命令具备统一 `--json`
- 非 TTY 不再发生隐式交互阻塞
- 核心错误具备稳定退出码
- 删除类命令使用统一确认策略
- 有正式 agent-friendly CLI 规范文档
- 有可追踪的 roadmap、milestone 和 issues

## 6. 非目标

本专项当前不建议纳入以下目标：

- 一次性重写所有命令输出
- 立即把所有写命令都做成完整 dry-run
- 在当前阶段承诺复杂交互式 TUI
- 在没有 CI 条件下先做过重的自动化框架

更现实的做法，是优先做统一底座和高收益命令。
