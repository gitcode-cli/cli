# 代码质量门禁规范

本文件定义 gitcode-cli 的代码质量门禁。

目的不是重复编码规范，而是明确“什么样的改动允许提交、进入 PR、合并到主线”。

## 1. 目标

代码质量门禁需要解决以下问题：

- 避免只有文档规范、没有执行标准
- 让多人和多 AI 客户端的交付标准一致
- 让本地检查、PR 审查和合并前确认形成闭环

## 2. 质量门禁分层

本仓库采用三级门禁：

1. 本地开发门禁
2. PR 门禁
3. 合并门禁

只有通过上一层门禁，才进入下一层。

## 3. 本地开发门禁

任何准备提交的改动，至少必须完成以下检查。

### 3.1 基础检查

```bash
go test ./...
go build -o ./gc ./cmd/gc
```

如果改动影响格式或存在格式风险，还应运行：

```bash
gofmt -w <files>
```

### 3.2 功能检查

如果改动影响具体命令行为，必须完成：

- 相关包级测试
- 至少一个真实命令验证

真实命令验证只能使用 `infra-test/*` 仓库。

### 3.3 安全检查

提交前必须确认：

- 无硬编码 token、password、secret
- 无敏感文件被误追踪
- 测试代码和文档不包含真实凭证

### 3.4 文档同步检查

行为变化后必须检查：

- `README.md`
- `docs/COMMANDS.md`
- `spec/*`
- `AGENTS.md`
- `CLAUDE.md`
- 相关 skills

### 3.5 工作区卫生检查

提交前必须确认：

- 构建产物未误提交
- 临时文件未误提交
- `.gitignore` 已覆盖新增本地产物

## 4. PR 门禁

进入 PR 前，必须满足以下要求。

### 4.1 关联与范围

- 改动有对应 issue
- 分支命名符合约定
- 变更范围清晰，不把无关内容混入同一个 PR

### 4.2 证据

PR 描述必须说明：

- 改动内容
- 测试情况
- 是否进行了真实命令验证
- docs-only 改动时是否未运行测试

### 4.3 审查信息

PR 提交后必须补：

- issue comment
- PR review comment

### 4.4 文档完整性

如果改动影响命令、流程、发布、审查或 AI 协作规则，PR 中必须体现同步文档更新。

## 5. 合并门禁

PR 在合并前，必须满足：

- 无 blocker 级问题
- 变更范围与 issue 一致
- 文档已同步
- 必要测试已完成
- 平台限制已明确写入文档，而不是隐藏或回避

在当前无 GitCode CI 环境的前提下，测试和审查证据以本地验证和人工确认作为主门禁。

## 6. 自动检查与人工审查边界

### 6.1 适合自动检查

- `go test ./...`
- `go build -o ./gc ./cmd/gc`
- 基础格式检查
- 常见敏感信息扫描
- 本地产物和临时文件检查

### 6.2 必须人工审查

- 命令行为是否符合产品语义
- 错误提示是否清晰
- API 兼容性是否真实有效
- 是否引入重复实现
- 文档是否准确描述当前限制
- AI 生成代码是否只是“看起来合理”

## 7. Blocker 判定

以下问题默认视为 blocker：

- 命令行为错误
- 参数解析回退或兼容性破坏
- 真实命令验证失败且无合理解释
- 敏感信息泄漏风险
- 文档与真实行为明显不一致
- 未处理的高优 review finding

以下问题通常不是 blocker，但应尽快修正：

- 轻微文案问题
- 非关键示例缺失
- 局部注释或表达不够清晰

## 8. docs-only 变更规则

如果 PR 是 docs-only 变更，可以不运行测试，但必须明确写出：

- 本次改动不涉及代码路径
- 未运行测试的原因
- 文档内容基于哪些现有实现或已验证行为

## 9. AI 协作补充规则

多人使用 Codex 和 Claude 时，质量门禁要求完全一致。

这意味着：

- `AGENTS.md` 和 `CLAUDE.md` 只能解释入口，不得降低门禁
- `.claude/skills/` 和 `.codex/skills/` 不能定义低于本文件的交付标准
- AI 生成的结果必须像人工提交一样接受同一套门禁

## 10. 当前执行基线

当前仓库在无 CI 环境下的执行基线为：

1. 本地开发门禁必须执行
2. PR 门禁必须完整记录
3. 合并门禁依赖人工审查和本地验证
4. 后续 CI 环境具备后，再将这些门禁映射进自动化
