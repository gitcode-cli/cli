# 评审流程

本文档定义作者记录、作者自检和独立执行主体评审的边界。

本文件的重点是避免 AI 或作者把“我已经检查过了”误写成“已经评审通过”。

## 职责

- 区分 issue 进度记录、作者自检、独立执行主体评审三类动作
- 定义独立执行主体评审的最小检查项
- 明确哪些结论可以推进状态，哪些不可以

## 流程概览

```
Issue 进度记录
→ PR 作者自检
→ 风险分级
→ 独立执行主体评审
→ 批准或要求继续修改
```

## 1. 三类记录的区别

### 1.1 Issue 进度记录

用途：

- 记录验证结果
- 记录修复进度
- 记录与 issue 相关的结论

它不是 PR 评审。

### 1.2 PR 作者自检

用途：

- 作者解释实现思路
- 作者提交测试与验证证据
- 作者说明风险与未覆盖项

它不是独立执行主体评审，不得直接把 PR 推到 `approved`。

### 1.3 独立执行主体评审

用途：

- 由非作者、非同一执行主体检查改动质量和风险
- 给出“通过”或“继续修改”的结论

独立执行主体可以是第二个 AI 代理，也可以是人工评审者。

高风险改动可以在独立 AI 评审后继续升级到人工最终确认。

只有独立执行主体评审才能把 PR 推进到 `status/approved` 或 `status/changes-requested`。

## 2. Issue 记录要求

### 评论时机

- issue 验证完成时
- 开发完成并进入 ready-for-review 前
- 出现阻塞或结论变化时

### Issue 记录模板

```markdown
## 阶段记录

- 阶段: verified / in-progress / blocked / ready-for-review
- 结论:
- 证据:
- 下一步:
```

## 3. PR 作者自检

### 评论时机

- 代码变更完成后
- 请求独立执行主体评审之前

### 评论内容

- 改动内容摘要
- 测试结果
- 实际命令验证结果
- 安全审查结果
- 文档同步结果
- 风险与未覆盖项

### 评论命令

```bash
gc pr review <number> --comment "## 作者自检

- 改动内容: ...
- 单元测试: ...
- 构建: ...
- 实际命令验证: ...
- 安全审查: ...
- 文档同步: ...
- 风险: ...
- 未覆盖项: ..." \
  -R gitcode-cli/cli
```

### 作者自检边界

- 作者自检不是批准
- 作者本人不得用自己的自检替代独立执行主体评审
- 自检完成后，PR 最多只能进入 `status/self-checked` 或 `status/ready-for-review`

## 4. 独立执行主体评审

### 评论时机

- PR 已完成作者自检
- PR 已进入 `status/ready-for-review`
- 已完成风险分级

### 评论内容

- 发现的问题或确认结果
- 是否存在 blocker
- 评审主体类型
- 评审主体标识
- 安全检查结论
- 是否建议继续修改

### 普通评审评论

```bash
gc pr review <number> --comment "## 评审结论

- 评审主体类型: AI / human
- 评审主体标识: codex-reviewer/session-456
- 发现: ...
- blocker: 无 / 有
- 安全检查: ...
- 结论: 可以批准 / 需要继续修改" \
  -R gitcode-cli/cli
```

### 批准

```bash
gc pr review <number> --approve -R gitcode-cli/cli
```

### 请求修改

```bash
gc pr review <number> --request -R gitcode-cli/cli
```

说明：GitCode 当前公开 API 暂不支持“request changes”动作；该命令会返回明确错误。当前执行规则是：

- 用 `gc pr review --comment` 留下结构化修改意见
- 同步把 PR 状态视为 `status/changes-requested`
- 待修改完成后重新进入 `status/ready-for-review`

### 4.1 风险分级与评审策略

- `low`：文档、规范、模板、共享 skill、入口文档等低风险改动，可由第二个 AI 代理完成独立执行主体评审并自动推进
- `medium`：一般命令实现、测试补齐、非认证类行为修正，可由第二个 AI 代理完成独立执行主体评审；只有在 blocker 或不确定时升级人工
- `high`：认证、token、权限、删除或覆盖、确认逻辑、JSON/schema/退出码契约、发布安装链路等高风险改动，独立 AI 评审后仍需人工最终确认

### 4.2 评审主体要求

- 评审主体必须与作者分离
- 评审主体不得与作者是同一执行主体
- 评审主体类型必须在评审记录中明确写为 `AI` 或 `human`
- 评审主体标识必须在评审记录中明确写出，且应可与作者主体标识区分
- 高风险改动的最终确认主体必须是人工

## 5. 独立执行主体评审检查清单

### 代码质量

- [ ] 代码符合编码规范
- [ ] 命名和结构清晰
- [ ] 无明显性能问题

### 安全检查

- [ ] 无硬编码凭证或敏感信息
- [ ] 认证、权限、配置读取逻辑未引入明显风险
- [ ] 删除、覆盖、远端写操作符合确认与非交互约束
- [ ] 未新增会导致自动化误操作的危险默认行为

### 行为正确性

- [ ] 命令行为符合 issue 和文档目标
- [ ] 错误提示清晰
- [ ] 非交互场景不会异常阻塞
- [ ] JSON 或 schema 输出未被破坏

### 测试与证据

- [ ] 有单元测试或合理说明
- [ ] 有构建结果
- [ ] 有实际命令验证或合理说明
- [ ] 有安全审查结果或合理说明
- [ ] 作者自检记录完整
- [ ] 风险分级结果已记录

### 文档同步

- [ ] `docs/COMMANDS.md` 已更新，或有无需更新说明
- [ ] `README.md` 已更新，或有无需更新说明
- [ ] `spec/*`、`AGENTS.md`、`CLAUDE.md`、相关 skills 已同步

### 评审主体独立性

- [ ] 评审主体与作者不是同一执行主体
- [ ] 评审主体类型已记录
- [ ] 评审主体标识已记录，且可与作者主体标识区分

## 6. 评审结果如何推进状态

- 只有作者自检：可到 `status/self-checked`
- 作者自检完成且请求评审：可到 `status/ready-for-review`
- 独立执行主体评审发现问题：到 `status/changes-requested`
- `low` / `medium` 风险改动经独立 AI 评审同意后：到 `status/approved`
- `high` 风险改动经独立 AI 评审后仍需人工最终确认，通过后到 `status/approved`

## 7. 检查清单

- [ ] issue 已有验证记录
- [ ] issue 已有开发进度记录
- [ ] PR 已有作者自检
- [ ] PR 已有独立执行主体评审结论
- [ ] 作者自检未被误当作批准
- [ ] 评审主体与作者不是同一执行主体
- [ ] 发现问题时已明确记录为 `changes-requested`
