# 评审流程

本文档定义作者记录、作者自检和多角色独立执行主体评审的边界与要求。

本文件的重点是避免 AI 或作者把"我已经检查过了"误写成"已经评审通过"。

## 职责

- 区分 issue 进度记录、作者自检、多角色独立执行主体评审三类动作
- 定义多角色评审的最小角色集和深度评审角色集
- 明确哪些结论可以推进状态，哪些不可以

## 流程概览

```
Issue 进度记录
→ PR 作者自检
→ 风险分级
→ 第一轮多角色评审（基础角色）
→ 发现问题则修复后第二轮评审
→ 无问题或第二轮评审通过
→ 批准或要求继续修改
```

## 1. 三类记录的区别

### 1.1 Issue 进度记录

用途：

- 记录验证结果
- 记录修复进度
- 记录与 issue 相关的结论

它不是 PR 评审。

### 1.2 PR 作者自检

用途：

- 作者解释实现思路
- 作者提交测试与验证证据
- 作者说明风险与未覆盖项

它不是独立执行主体评审，不得直接把 PR 推到 `approved`。

### 1.3 多角色独立执行主体评审

用途：

- 由多个独立 Agent 分角色检查改动质量和风险
- 每个角色专注于特定评审维度
- 综合各角色结论给出最终评审结果

独立执行主体必须是多个非作者、非同一执行主体的 Agent。

高风险改动可以在多角色 AI 评审后继续升级到人工最终确认。

只有多角色独立执行主体评审才能把 PR 推进到 `status/ready-for-review`、`status/approved` 或 `status/changes-requested`。

## 2. 多角色评审要求

### 2.1 第一轮评审（基础角色，必须执行）

提交 PR 前必须完成以下 **4 个基础评审角色**：

| 角色 | 职责 | 检查要点 |
|------|------|----------|
| **代码审查** | 检查代码实现质量 | 代码逻辑、项目模式遵循、命名规范、错误处理 |
| **安全审查** | 检查安全性 | 凭证泄漏、Token 处理、API 安全、危险写路径 |
| **测试审查** | 检查测试覆盖 | 测试场景覆盖、测试质量、Mock 使用正确性 |
| **文档审查** | 检查文档同步 | COMMANDS.md 更新、示例正确、与代码一致性 |

**触发条件**：PR 创建后、进入 ready-for-review 前必须执行第一轮评审。

#### 2.1.1 docs-only PR 例外处理

对于纯文档更新的 PR，可使用简化评审路径：

**docs-only PR 判定标准**：
- 仅修改 `docs/`、`README.md`、`CONTRIBUTING.md`、`spec/` 等文档/规范文件
- 无代码文件改动（排除 `.go`、`.py`、`.sh` 等代码文件）
- 无测试文件改动

**简化评审路径**：

| 角色 | 是否需要 | 理由 |
|------|----------|------|
| 代码审查 | 不需要 | 无代码改动 |
| 安全审查 | **必须** | 检查文档是否误写真实凭证 |
| 测试审查 | 不需要 | 无测试文件 |
| 文档审查 | **必须** | 核心评审内容 |

docs-only PR 仅需执行 **2 个评审角色**（文档审查 + 安全审查）。

**执行方式**：
```bash
# 创建评审团队
TeamCreate

# 仅启动 2 个评审 Agent
Agent(description="文档审查", subagent_type="general-purpose")
Agent(description="安全审查", subagent_type="general-purpose")
```

#### 2.1.2 trivial PR 例外处理

对于极简单的 PR（如修复拼写错误、单行修改），可使用简化评审路径：

**trivial PR 判定标准**：
- 改动行数 ≤ 5 行
- 无逻辑变更（仅格式、拼写、注释）
- 无新增功能

**与 docs-only PR 的区别**：
- docs-only PR：仅文档文件改动
- trivial PR：可含代码改动（如单行拼写修复），但逻辑不变

trivial PR 仍需执行简化评审路径（**文档审查 + 安全审查**），或由人工确认后直接批准（需记录人工批准理由）。

### 2.2 第二轮评审（深度角色，发现问题时执行）

第一轮发现问题修复后，或高风险改动，应执行以下 **4 个深度评审角色**：

| 角色 | 职责 | 检查要点 |
|------|------|----------|
| **架构一致性** | 检查架构遵循 | Options 结构体、函数命名、Factory 使用、与其他命令对比 |
| **API 契约** | 检查 API 调用 | API 参数正确、调用时序、错误处理完整性 |
| **边界条件** | 检查边界处理 | 输入验证、空值处理、网络错误、文件操作边界 |
| **用户体验** | 检查命令友好性 | 帮助文本、错误消息、输出格式、参数设计 |

**触发条件**：
- 第一轮评审发现问题需要修复后重新评审
- `risk/high` 改动必须执行第二轮评审
- 复杂改动（跨多文件、新命令实现）建议执行第二轮评审

### 2.3 评审执行方式

```bash
# 使用 TeamCreate 创建评审团队
# 启动多个 Agent 并行执行评审
# 每个 Agent 专注于一个评审角色
```

评审 Agent 必须：
- 与作者 Agent 是不同的执行主体
- 有明确的角色标识（如 `code-reviewer`, `security-reviewer`）
- 输出结构化的评审结论

### 2.4 评审结论格式

每个角色输出统一格式：

```markdown
## [角色名称]评审结论

### 检查项
| 检查项 | 结果 |
|--------|------|
| ... | ✓/✗ |

### 发现问题
- [如有问题列出，没有写"无"]

### 建议
- [可选的改进建议]

### 总体评价
- [approved / needs-improvement / blocked]
```

## 3. Issue 记录要求

### 评论时机

- issue 验证完成时
- 开发完成并进入 ready-for-review 前
- 出现阻塞或结论变化时

### Issue 记录模板

```markdown
## 阶段记录

- 阶段: verified / in-progress / blocked / ready-for-review
- 结论:
- 证据:
- 下一步:
```

## 4. PR 作者自检

### 评论时机

- 代码变更完成后
- 启动多角色评审之前

### 评论内容

- 改动内容摘要
- 测试结果
- 实际命令验证结果
- 安全审查结果
- 文档同步结果
- 风险与未覆盖项

### 作者自检边界

- 作者自检不是批准
- 作者本人不得用自己的自检替代多角色评审
- 自检完成后，PR 最多只能进入 `status/self-checked`
- 多角色评审完成后才能进入 `status/ready-for-review` 或 `status/approved`

## 5. 多角色评审流程

### 5.1 流程步骤

```
1. 作者完成自检
2. 作者创建评审团队（TeamCreate）
3. 启动第一轮 4 个基础评审 Agent
4. 收集各角色评审结论
5. 如发现问题：
   a. 作者修复问题
   b. 启动第二轮 4 个深度评审 Agent
   c. 收集第二轮评审结论
6. 如仍有问题，重复修复和评审
7. 所有角色通过后，汇总评审结论
8. 推进 PR 状态
```

### 5.2 评审汇总格式

```markdown
## 多角色独立执行主体评审汇总

### 第一轮评审结论

| 评审角色 | 结论 | 主要发现 |
|----------|------|----------|
| 代码审查 | approved / needs-improvement | ... |
| 安全审查 | approved / blocked | ... |
| 测试审查 | approved / needs-improvement | ... |
| 文档审查 | approved / needs-update | ... |

### 第二轮评审结论（如有）

| 评审角色 | 结论 | 主要发现 |
|----------|------|----------|
| 架构一致性 | approved / needs-discussion | ... |
| API 契约 | approved / needs-optimization | ... |
| 边界条件 | approved / needs-improvement | ... |
| 用户体验 | approved / needs-improvement | ... |

### 发现问题及修复情况

| 问题 | 修复提交 | 验证 |
|------|----------|------|
| ... | commit hash | ✓ |

### 后续跟踪 Issues（如有）

| Issue | 问题 | 类型 |
|-------|------|------|
| ... | ... | ... |

### 总体评审结论

- [approved / changes-requested]
- 评审执行主体：N 个独立 Claude 子 Agent

评审角色：[列出参与评审的角色]
```

### 5.3 风险分级与评审策略

| 风险等级 | 第一轮评审 | 第二轮评审 | 最终确认 |
|----------|------------|------------|----------|
| `low` | 必须 | 可选 | AI 汇总即可 |
| `medium` | 必须 | 发现问题时执行 | AI 汇总即可 |
| `high` | 必须 | 必须 | 人工最终确认 |

## 6. 各评审角色检查清单

### 6.1 代码审查

- [ ] 代码逻辑正确
- [ ] 遵循项目现有模式（对比同类命令）
- [ ] 命名和结构清晰
- [ ] 错误处理完整
- [ ] 无废弃 API 使用
- [ ] 无明显性能问题

### 6.2 安全审查

- [ ] 无硬编码 Token/密钥
- [ ] Token 通过环境变量获取
- [ ] 测试代码不含真实 Token
- [ ] 使用认证客户端
- [ ] 无危险写路径（或正确处理）
- [ ] 文档不含真实凭证
- [ ] 测试仓库使用 infra-test/*

### 6.3 测试审查

- [ ] 测试覆盖正常流程
- [ ] 测试覆盖错误场景
- [ ] 测试命名清晰
- [ ] 测试独立、可重复
- [ ] Mock 使用正确
- [ ] 边界条件测试覆盖

### 6.4 文档审查

- [ ] docs/COMMANDS.md 已更新
- [ ] 示例命令正确且完整
- [ ] --json 支持列表已更新（如有）
- [ ] flags 与文档描述一致
- [ ] 说明清晰准确

### 6.5 架构一致性

- [ ] Options 结构体字段命名一致
- [ ] 函数命名遵循项目模式（NewCmdXxx, xxxRun）
- [ ] Factory 使用正确（IOStreams, HttpClient, BaseRepo）
- [ ] 与同类命令对比一致
- [ ] JSON 输出支持一致

### 6.6 API 契约

- [ ] API 调用参数正确
- [ ] API 调用时序合理
- [ ] API 错误处理完整
- [ ] 使用 %w 包装错误链

### 6.7 边界条件

- [ ] 输入验证完整（格式、范围）
- [ ] 空值处理正确
- [ ] 网络错误处理
- [ ] 文件操作边界检查
- [ ] 不存在资源处理
- [ ] 无效输入提示友好

### 6.8 用户体验

- [ ] 帮助文本完整覆盖场景
- [ ] 示例命令可直接复制使用
- [ ] 错误消息提供解决方案
- [ ] 输出格式一致
- [ ] 参数设计符合直觉

## 7. 评审结果如何推进状态

- 只有作者自检：可到 `status/self-checked`
- 第一轮评审发现问题：保持 `status/self-checked`，修复后重新评审
- 第一轮评审全部通过（low/medium 风险）：可到 `status/approved`
- 第一轮通过但 high 风险：等待人工最终确认后到 `status/approved`
- 第二轮评审发现问题：修复后重新评审
- 第二轮评审全部通过：可到 `status/approved`（high 风险需人工确认）
- 评审主体发现 blocker：到 `status/changes-requested`

## 8. 检查清单

### 8.1 标准 PR 检查清单

- [ ] issue 已有验证记录
- [ ] issue 已有开发进度记录
- [ ] PR 已有作者自检
- [ ] PR 已完成第一轮 4 角色评审
- [ ] 发现问题已完成第二轮 4 角色评审（如有）
- [ ] 所有评审角色结论为 approved
- [ ] 作者自检未被误当作批准
- [ ] 评审 Agent 与作者 Agent 不是同一执行主体
- [ ] 发现问题时已创建 Issue 跟踪（非阻塞问题）
- [ ] 评审汇总评论已添加到 PR

### 8.2 docs-only PR 检查清单

- [ ] 仅修改文档文件（无 .go/.py/.sh 等代码文件）
- [ ] PR 已有作者自检
- [ ] PR 已完成文档审查 + 安全审查（2 角色）
- [ ] 安全审查确认无真实凭证泄漏
- [ ] 文档审查确认示例正确、格式一致
- [ ] 评审汇总评论已添加到 PR

## 9. 禁止事项

- 禁止跳过第一轮基础评审直接提交 PR
- 禁止用作者自检替代多角色评审
- 禁止评审 Agent 与作者 Agent 是同一执行主体
- 禁止声称"已评审通过"但缺少结构化评审结论
- 禁止发现 P0 问题后直接合并而不修复
- 禁止隐瞒评审发现的问题

## 10. Loop Engineering 评审边界

Loop Engineering 中，作者自检不能替代独立评审。

独立评审必须能看到：

- 关联 issue / PR
- diff 摘要
- self-check
- 测试和真实命令验证
- GitHub mirror CI evidence
- 风险和未覆盖项

评审结论必须写回 GitCode PR。若评审使用 Claude Dynamic Workflows 或 agent team 并行执行，最终结论仍必须受 `spec/`、GitCode PR 状态和 CI evidence 约束。
